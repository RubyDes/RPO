**Дата:** 06 декабря 2025 г.

**Статус:** Принято

**Контекст:**

Для платформы онлайн-обучения необходимо выбрать архитектурную форму, которая обеспечит масштабируемость для тысяч одновременных пользователей, высокую доступность сервиса и возможность независимого развития различных модулей (управление курсами, тестирование, видеоконтент, коммуникации).

**Рассмотренные варианты:**

1. **Модульный монолит:** Единое приложение с четко разделенными модулями.
2. **Микросервисная архитектура:** Независимые сервисы для каждой функциональной области.
3. **Серверлесс архитектура:** Функции как сервис с автоматическим масштабированием.

**Решение:**

Выбрана **микросервисная архитектура** с элементами модульного подхода.

**Обоснование:**

- **Независимое масштабирование:** Сервис видеоконтента и тестирования можно масштабировать независимо от пользовательского сервиса. (Обеспечивает SCA01-SCA04, AVA01, PRN02)
- **Высокая доступность:** Отказ одного сервиса не приводит к полной недоступности платформы. (Обеспечивает AVA01, DUR04, DUR05)
- **Независимые циклы разработки:** Команды могут работать над разными сервисами параллельно. (Обеспечивает ускорение разработки в рамках LIM03)
- **Технологическая гибкость:** Возможность использовать разные технологии для разных сервисов (например, специализированные решения для видеообработки). (Обеспечивает PRN02, гибкость в рамках LIM02)

**Последствия:**

- Необходимо внедрить API Gateway для маршрутизации запросов.
- Требуется надежный механизм межсервисной коммуникации (синхронной и асинхронной).
- Увеличивается сложность мониторинга и отладки распределенной системы.
